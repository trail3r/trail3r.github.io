# 주석 작성 규칙

이 문서는 템플릿을 사용하는 사람이 소스의 주석만 읽어도 기능의 사용법과 수정 지점을
이해할 수 있도록 주석 작성 방식을 정의합니다.

## 적용 범위

다음 경로에서 직접 작성하고 관리하는 파일에 적용합니다.

- "_includes/customs/"
- "_sass/customs/"
- "assets/js/customs/"

테마 원본, 외부 라이브러리, 자동 생성 파일에는 적용하지 않습니다. 외부 코드에 포함된
라이선스와 저작권 고지는 수정하거나 제거하지 않습니다.

## 기본 원칙

- 주석은 한국어로 작성합니다.
- 문장은 존댓말인 "합니다." 체로 끝냅니다.
- 변수, 클래스, ID, 설정값, 파일 경로처럼 코드와 정확히 대응하는 명칭은 큰따옴표로
  감쌉니다.
- 코드 예제 안에서는 명칭을 구분하기 위한 큰따옴표를 추가하지 않습니다.
- 코드만 읽어도 알 수 있는 내용을 주석으로 반복하지 않습니다.
- 과거 변경 내역, 코드 작성자, 코드 생성 도구는 주석이 아니라 Git 기록에서 관리합니다.

## 파일 헤더

직접 관리하는 파일에는 최소 한두 문장의 헤더 주석을 작성합니다. 헤더의 첫 문단은
"역할"이나 "목적" 같은 표제를 붙이지 않고 자연스러운 문장으로 파일의 기능을 설명합니다.

사용자에게 추가 정보가 필요한 경우에만 다음 항목을 순서대로 작성합니다.

1. 사용법
2. 설정
3. 관련 파일
4. 주의사항

내용이 없는 항목은 만들지 않습니다. 현재 파일의 이름과 경로는 편집기에서 확인할 수 있고
파일을 이동하면 오래된 정보가 되므로 헤더에 작성하지 않습니다. 함께 확인하거나 수정해야
하는 다른 파일의 경로는 "관련 파일"에 작성할 수 있습니다.

### 위치

헤더는 파일의 맨 위에 작성합니다. 다음 내용이 반드시 먼저 나와야 하는 파일에서는 해당
내용 바로 아래에 헤더를 작성합니다.

- Jekyll의 YAML Front Matter
- 기존 라이선스 또는 저작권 고지

### 주석 문법

Liquid 파일은 빌드 결과에 설명이 출력되지 않도록 "{% comment %}"를 사용합니다.

```liquid
{% comment %}
홈페이지에 대표 게시글을 카드 형태로 표시합니다.
{% endcomment %}
```

JavaScript 파일은 블록 주석을 사용합니다.

```javascript
/*
홈페이지의 대표 게시글 목록에 캐러셀 동작을 추가합니다.
*/
```

SCSS 파일은 컴파일된 CSS에 설명이 포함되지 않도록 "//"를 사용합니다.

```scss
// 게시글 페이지에 표시되는 목차의 디자인을 정의합니다.
// 목차 구조를 변경할 때는 관련 Liquid 파일도 함께 확인합니다.
```

긴 등호나 하이픈으로 만든 장식용 구분선은 사용하지 않습니다.

## 사용법

사용자가 파일을 직접 호출하거나 설정해야 한다면 복사해서 바로 사용할 수 있는 최소 예제를
제공합니다.

- Liquid include에는 실제 "{% include %}" 예제를 작성합니다.
- JavaScript에는 동작에 필요한 HTML 클래스나 "data-*" 속성을 설명합니다.
- SCSS에는 필요한 import 또는 변경할 수 있는 변수를 설명합니다.
- 자동으로 불러오는 파일에는 예제를 만들지 않고, 어디에서 불러오며 언제 동작하는지
  설명합니다.

```liquid
{% comment %}
홈페이지에 대표 게시글을 카드 형태로 표시합니다.
게시글의 Front Matter에 "featured: true"를 추가하면 목록에 포함됩니다.

사용법
    {% include customs/pages/home/featured.html limit=5 %}

설정
    "limit"은 표시할 게시글의 최대 개수이며 기본값은 5입니다.

관련 파일
    - "_sass/customs/pages/home/_featured.scss"
    - "assets/js/customs/pages/home/featured.js"
{% endcomment %}
```

## 코드 중간 주석

코드 중간 주석은 구현의 동작을 다시 설명하는 대신 코드만으로 파악하기 어려운 이유와 제약을
설명합니다.

- 해당 구현을 선택한 이유
- 쉽게 놓칠 수 있는 예외 상황
- 성능, 접근성 또는 브라우저 호환성 제약
- 수정할 때 함께 지켜야 하는 조건
- 코드만 봐서는 알 수 없는 부작용

```javascript
// 이미지가 로드되기 전에는 너비 계산이 부정확하므로 탐색을 잠시 막습니다.
button.disabled = true;
```

"버튼을 비활성화합니다.", "변수를 초기화합니다."처럼 바로 아래의 코드를 그대로 풀어 쓴
주석은 작성하지 않습니다.

## 함수 주석

모든 함수에 설명 주석을 의무적으로 작성하지 않습니다. 함수의 이름과 코드만으로 역할이
분명하면 별도의 주석을 작성하지 않습니다.

다음과 같은 경우에는 함수 위에 설명을 추가합니다.

- 호출 방법이나 반환값이 예상과 다릅니다.
- 외부에서 재사용하는 함수입니다.
- 중요한 전제 조건이나 부작용이 있습니다.
- 브라우저 API의 특이한 동작을 보완합니다.

## 미완성 작업과 알려진 문제

"TODO"와 "FIXME" 대신 "할 일"과 "수정 필요"를 사용합니다. 표시만 남기지 않고 무엇을
해야 하는지 구체적인 문장으로 작성합니다.

```javascript
// 할 일: 키보드의 좌우 방향키로도 카드를 이동할 수 있게 합니다.

// 수정 필요: 빠르게 크기를 변경하면 버튼 상태가 늦게 갱신됩니다.
```

## 유지보수

코드 변경으로 동작, 설정 또는 의존성이 달라지면 같은 작업에서 관련 주석도 함께 수정합니다.
코드와 맞지 않는 주석은 코드 오류와 동일하게 취급합니다.

다음 내용을 변경할 때는 관련 주석을 반드시 확인합니다.

- 옵션 또는 기본값
- 클래스, ID 또는 파일 경로
- 관련 파일
- 예외 처리 또는 브라우저 지원 범위
- 기능의 추가 또는 제거
